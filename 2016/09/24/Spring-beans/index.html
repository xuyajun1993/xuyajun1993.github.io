Spring-beans

发表于 2016-09-24   |   分类于 Spring源码   |     |   阅读次数

       首先感谢《spring源码深度解析》这本书，因为知道已经有前辈整理了阅读经验，让我更加能够坚持将源码进行到底。但为了达到锻炼自己最大化，我还是决定不直接去读，先按照自己的思路来。只有在不知道怎么继续下去的时候才借鉴一下前辈的经验。

先做点准备工作：

  1. 下载spring源码，选择的是3.2.15版本。
  2. 因为spring已经由maven转向gradle,所以需要较为深刻的了解下gradle。通过gradle查看依赖树，从整体上弄清每个模块的关系和功能。
  3. gradle编译源码遇到问题：”A problem occurred configuring project ‘:spring-orm-hibernate4’”，通过Google，这个貌似是3.2版本和gradle版本的不兼容造成的。先不折腾，注释掉merge.into语句。
  4. 将源码文件夹改为只读，防止不小心修改
    chmod -R 555 spring-framework-3.2.15.RELEASE

Bean:

       开始肯定是先找到一个切入类，我的做法是快速浏览一下项目的结构，熟悉每个package大概是什么功能。比如Spring-beans结构由annotation，factory，propertyeditors，support几个包组成，还有很多未放入包的类，不过，一看类名就知道大概是工具类和自定义的异常类。至于几个package，annotation里面定义了一些注解，propertyeditors定义了数据类型之间的转换，support提供了对bean排序等的一些支持功能，继续展开factory，终于找到一个熟悉的类XmlBeanFactory。
       相信大家刚开始学习spring的时候肯定写过这行代码：
BeanFactory factory = new XmlBeanFactory(new ClassPathResource("applicationContext.xml"));
       打开发现该类已经@Deprecated，谷歌了一下，和猜想的一样，3.1版本已经被applicationcontext替换。
BeanFactory factory = new ClassPathXmlApplicationContext("applicationContext.xml");
而applicationcontext属于context模块下，但还是基于beans模块实现的。不管，先研究下XmlBeanFactory。

创建bean

用idea显示出XmlBeanFactory的类图：

ok,从BeanFactory入手，看看bean怎么被生产的，方法如图：典型的工厂模式，这个就是创建bean的顶级工厂。
BeanFactory有三个直接子类，也都是接口，初步看了一下，对应的功能分别如下:

  • ListableBeanFactory: 可批量创建bean，获取bean信息
  • HierarchicalBeanFactory: 使beanfactory具有继承性
  • AutowireCapableBeanFactory: 定义了自动装配的策略

再看下类图左边的几个接口，容易理解。

  • AliasRegistry: 提供了别名管理
  • BeanDefinitionRegistry: AliasRegistry的子接口，主要用来注册bean
  • SimpleAliasRegistry: AliasRegistry接口的简单实现类，使用key和value都是String的map对别名进行存储，删除等操作。
  • SingletonBeanRegistry: 另一个顶级接口,用来注册singleton的bean。

类图中其他类都是由这几个接口衍生出来，但易看出，所有的接口最后都在DefaultListableBeanFactory类中实现。先不细看这一千多行代码，初步了解了beanfactory的继承图和功能就行。

定义bean

那接下来看啥，嘿嘿，看Bean怎么被定义的。在BeanDefinitionRegistry中注册bean时，使用了BeanDefinition。
找到其一个子类RootBeanDefinition来看看类图：
看下各个接口的功能:

  • AttributeAccessor: 定义bean的属性（字段)
  • BeanMetadataElement: 只有一个getSource()方法，返回元数据的配置源
  • BeanDefinition: 定义了bean的环境属性，比如single/prototype，是否懒加载
  • AttributeAccessorSupport: 提供了AttributeAccessor接口的简易实现，存储/删除/判断bean的属性（字段）
  • BeanMetadataAttributeAccessor: AttributeAccessorSupport类的增加版，用BeanMetadataAttribute对象代替了Object
  • AbstractBeanDefinition:抽象类，为GenericBeanDefinition，RootBeanDefinition，ChildBeanDefinition三个子类抽象出公共的方法和属性
  • ChildBeanDefinition: 让bean拥有继承关系，拥有父bean
  • RootBeanDefinition: 可以看作是多个具有继承关系bean的合成体
  • GenericBeanDefinition:2.5版本后大部分场景下用来代替ChildBeanDefinition和RootBeanDefinition，后两个类更倾向于使用在提前决定了继承关系的场景中，而GenericBeanDefinition可以动态配置继承关系

关于bean的定义大致如此，先了解到这，来看下bean的解析。

bean的解析

可以预见，内部的某个reader读取spring配置文件，然后解析成BeanDefinition对象。在XmlBeanFactory类中发现XmlBeanDefinitionReader，看下类图：

  • AbstractBeanDefinitionReader: 抽象类，实现了BeanDefinitionReader接口，为PropertiesBeanDefinitionReader和XmlBeanDefinitionReader子类抽象出公共的方法和属性*

为了更好理解这几个类，需要了解下spring中内部对资源的处理，虽然是core模块中实现的。

  • InputStreamSource: 该接口只有一个getInputStream方法
  • Resource: 将输入流包装成资源
  • AbstractResource: 抽象类，继承Resource接口，进行了简单实现，不同来源的资源文件都会继承该基类进行各自的实现
  • WritableResource: 增加资源的写操作
  • AbstractFileResolvingResource: 将url形式的输入流转为文件引用输入流

搞清楚Resource后，来梳理一下bean的解析过程(XmlBeanDefinitionReader.loadBeanDefinitions):

  1. 封装资源:
    • 从XmlBeanFactory中传到XmlBeanDefinitionReader类中的是Resource对象，先封装为EncodedResource，指定编码
  2. 数据检验和准备:
    • 使用了一个ThreadLocal保存EncodedResource，如果重复读取某个xml配置文件，会报错
    • 从EncodedResource获取输入流，构建成org.xml.sax.InputSource
  3. 真正解析xml:
    • 获取对XML文件的验证模式
    • 使用DefaultDocumentLoader加载XML文件，得到org.w3c.dom.Document对象。
    • 根据Document对象和资源注册bean

       问题来了,看到个不太懂得玩意，对XML文件的验证模式？看了下代码，如果XML文件中包含“DOCTYPE”，则使用DTD进行验证，否则使用XSD。DTD和XSD有啥区别呢？请移步另一片博文：DTD
       关于通过JAXP的dom接口(《spring源码深度解析》里作者说是使用SAX接口来解析的，需要纠正)来解析XML文档，需要了解下spring实现了ResourceEntityResolver和DelegatingEntityResolver两种EntityResolver。简单了解下EntityResolver: 解析一个XML时，需要根据开头的声明去网络下载DTD文件，对XML进行格式验证。而重写EntityResolver就可以指定去本地寻找DTD文件，避免网络操作。
       重点放在注册bean上，由于spring很好的践行了单一职责设计模式，所以类的跳转很多。先来到DefaultBeanDefinitionDocumentReader.doRegisterBeanDefinitions:

  1. 处理“profile”属性：

    • 关于profile属性，官方给出的例子如下： 

      <beans profile="dev">
    <jdbc:embedded-database id="dataSource">
        <jdbc:script location="classpath:com/bank/config/sql/schema.sql"/>
        <jdbc:script location="classpath:com/bank/config/sql/test-data.sql"/>
    </jdbc:embedded-database>
</beans>

<beans profile="production">
    <jee:jndi-lookup id="dataSource" jndi-name="java:comp/env/jdbc/datasource"/>
</beans>

      如果项目中使用dev环境下的数据源，则

      AnnotationConfigApplicationContext ctx = new AnnotationConfigApplicationContext();
ctx.getEnvironment().setActiveProfiles("dev");
ctx.register(SomeConfig.class, StandaloneDataConfig.class, JndiDataConfig.class);
ctx.refresh();

      所以如果配置了profile属性，则需要检验环境变量中的profile是否在配置中定义了，如果没有，则不进行下面的解析。

  2. 委托BeanDefinitionParserDelegate解析

    • 核心代码：

      protected void parseBeanDefinitions(Element root, BeanDefinitionParserDelegate delegate) {
      if (delegate.isDefaultNamespace(root)) {
          NodeList nl = root.getChildNodes();
          for (int i = 0; i < nl.getLength(); i++) {
              Node node = nl.item(i);
              if (node instanceof Element) {
                  Element ele = (Element) node;
                  if (delegate.isDefaultNamespace(ele)) {
                      parseDefaultElement(ele, delegate);
                  }
                  else {
                      delegate.parseCustomElement(ele);
                  }
              }
          }
      }
      else {
          delegate.parseCustomElement(root);
      }
  }

    可以看出，如果dom节点的名称空间是spring默认的“http://www.springframework.org/schema/beans”，则直接按照默认标签解析。否则，就需要按照自定义的解析。
